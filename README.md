# Ternak Klip · Inventory Dashboard

A read-only inventory dashboard for Ternak Klip's Operations Department. Displays product stock levels, surfaces low-stock alerts, and lets operators refresh data on demand. Built as an MVP case study — no authentication, no CRUD, no backend layer.

---

## Tech Stack

| Layer        | Choice                                    |
| ------------ | ----------------------------------------- |
| Framework    | React 18 + Vite 6                         |
| Language     | TypeScript (strict mode, no `any`)        |
| Styling      | Tailwind CSS v4 (via `@tailwindcss/vite`) |
| Database     | Supabase (PostgreSQL)                     |
| Data client  | `@supabase/supabase-js`                   |
| Tooling      | ESLint + Prettier                         |
| Deploy target| Vercel (static export)                    |

## Project Structure

```
src/
├── components/          # Feature components
│   ├── ui/              # Design-system primitives (Badge, Button, Card, Skeleton)
│   ├── DashboardHeader.tsx
│   ├── StatsCards.tsx
│   ├── ProductTable.tsx
│   ├── StockBadge.tsx
│   ├── DashboardSkeleton.tsx
│   ├── EmptyState.tsx
│   ├── ErrorState.tsx
│   └── EnvMissing.tsx
├── hooks/
│   └── useProducts.ts   # Fetching / loading / error / refetch state
├── lib/
│   ├── supabase.ts      # Client singleton + env validation
│   └── utils.ts         # Date formatters, stock-status helper, cn()
├── types/
│   └── product.ts
├── App.tsx
├── main.tsx
└── index.css            # Tailwind v4 theme tokens + base styles
```

**Reasoning:** The structure separates concerns so each directory has one job. `components/ui/` holds stateless primitives that can be composed anywhere. Feature components like `ProductTable` compose primitives and receive data via props — they never fetch. All fetching lives in `hooks/useProducts.ts`, all formatting in `lib/utils.ts`. This keeps components small (<80 LOC) and testable.

## Setup

### 1. Clone and install

```bash
git clone <repo-url>
cd inventory
npm install
```

### 2. Configure environment

Copy the example and fill in your Supabase credentials:

```bash
cp .env.example .env
```

Edit `.env`:

```
VITE_SUPABASE_URL=https://your-project-ref.supabase.co
VITE_SUPABASE_ANON_KEY=your-supabase-anon-key
```

> If either variable is missing, the app will render a clear error screen at startup instead of silently failing.

### 3. Set up the Supabase database

Run this SQL in the Supabase SQL Editor:

```sql
CREATE TABLE products (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  stock_count INT NOT NULL DEFAULT 0,
  last_updated TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now())
);

INSERT INTO products (name, stock_count)
VALUES
  ('T-Shirt - Small', 50),
  ('Jeans - Blue', 25),
  ('Sneakers - Black', 12);

-- Allow anonymous reads (the app has no auth)
ALTER TABLE products ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Allow anon read" ON products FOR SELECT TO anon USING (true);
```

### 4. Run the dev server

```bash
npm run dev
```

Open http://localhost:5173.

## Design Decisions

### Why Vite over Next.js

Next.js would be overkill here. This is an internal SPA — no SEO, no SSR, no routing, no serverless functions. Vite gives us fast HMR, a tiny build, and static deployment to Vercel without paying the complexity tax of an SSR framework.

### Why connect directly to Supabase from the frontend

The case study is explicit: MVP scope, read-only, no auth. Adding a backend API would be a wrapper around a single `SELECT * FROM products` query with zero added value. Supabase's row-level security + anon key already enforces read-only access at the database layer, which is the correct place for that boundary. If the product later needed writes, mutations, or per-user scoping, that's when a backend would earn its keep.

### Stock-level color coding

| Range              | Status   | Color | Rationale                                            |
| ------------------ | -------- | ----- | ---------------------------------------------------- |
| `>= 30`            | Healthy  | Green | Comfortable buffer — no action needed                |
| `>= 10` and `< 30` | Warning  | Amber | Restock window — plan next order                     |
| `< 10`             | Critical | Red   | Immediate attention — risk of stock-out this week    |

The thresholds are defined in `src/lib/utils.ts` (`getStockStatus`) so they can be tuned centrally without touching any component.

### Design direction

The UI deliberately avoids "AI slop" patterns: no purple/violet gradients, no generic `#3B82F6` blue, no Inter. The palette is a warm off-white canvas (`#FAFAF9`), near-black ink text (`#0B1220`), and a single teal accent (`#0F766E`) for CTAs. Semantic green/amber/red is reserved for stock status only. Typography is Plus Jakarta Sans throughout, with weight and size doing the hierarchy work. Each table row has a 3px left accent bar matching its stock status — scannable at a glance, no visual noise.
